# Обмен контекстом без облака и без git

## Что устанавливаем

Обязательное:

- `Syncthing` на каждую машину аналитика
- `Docker Desktop`
- `Ollama`
- `VS Code`
- `Continue`

Для сравнения конфликтных версий рекомендуем не отдельный diff-инструмент, а встроенный `VS Code Compare`.

### Где брать Syncthing

- официальная страница загрузки: [syncthing.net/downloads](https://syncthing.net/downloads/)
- macOS app bundle: [syncthing-macos](https://github.com/syncthing/syncthing-macos)
- Windows installer: [Syncthing Windows Setup](https://github.com/Bill-Stewart/SyncthingWindowsSetup)

Сам бинарник `Syncthing` в проект не вшиваем. Причина простая: это отдельное приложение со своим release cycle, и для on-prem установки надёжнее ставить его с официального источника прямо на машину аналитика.

## Как устроен обмен

У каждого аналитика проект и модели остаются локально. Между аналитиками синхронизируется только отдельная папка обмена, например:

```text
team-exchange/
  bundles/
```

Внутри `bundles/` система публикует неизменяемые пакеты с `manifest.json` и файлами общего контекста:

- `docs/shared-context`
- `docs/templates`
- `docs/glossary`

## Как настроить

1. Установи `Syncthing` на каждую машину.
2. На обеих машинах создай или выбери одну и ту же логическую папку обмена, например:
   - macOS: `/Users/<user>/team-exchange`
   - Windows: `C:\\team-exchange`
3. Запусти `Syncthing` на машине A и на машине B и открой его web-интерфейс.
4. На машине A скопируй её `Device ID`. На машине B скопируй её `Device ID`.
5. На машине A нажми `Add Remote Device`, вставь `Device ID` машины B и сохрани.
6. На машине B сделай то же самое для машины A.
7. Когда устройства добавлены друг к другу, на одной из машин нажми `Add Folder`.
8. Укажи `Folder Path` до папки обмена и в `Share With` выбери вторую машину.
9. На второй машине подтверди входящую папку и укажи локальный путь, где она должна лежать.
10. Дождись статуса `Up to Date` на обеих машинах. Для быстрой проверки можно положить в папку тестовый файл `sync-test.md` и убедиться, что он появился на второй стороне.
11. В `Подготовка окружения` укажи путь к папке обмена.
12. Нажми `Сохранить настройки`.
13. Перезапусти стек через `./start.command`, чтобы Docker смонтировал новую папку обмена внутрь сервиса.

## Как работать дальше

1. Положи общий контекст проекта в локальные папки:
   - `docs/shared-context`
   - `docs/templates`
   - `docs/glossary`
2. На странице `Обмен контекстом` нажми `Опубликовать свои изменения`.
3. У коллег появятся новые bundle-пакеты.
4. Они открывают страницу `Обмен контекстом` и нажимают `Забрать к себе`.

## Что происходит при конфликте

Если локальный файл уже существует и отличается от входящего, система не перезатирает его молча. Вместо этого рядом создаётся incoming-версия:

```text
example.incoming_<bundle-id>.md
```

Рекомендуемый сценарий:

1. открыть локальный файл и incoming-версию в `VS Code Compare`
2. перенести нужные правки
3. удалить incoming-файл после разборки

## Что не синхронизируем

- логины и пароли
- `Playwright` state
- `.continue/config.yaml`
- локальные модели `Ollama`
- кэши, `node_modules`, `.venv`
- временные рабочие артефакты
