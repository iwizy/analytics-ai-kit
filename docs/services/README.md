# База контекста по микросервисам

В этой папке лежит service-specific контекст, который система использует:

- при сборке `draft`
- при `gaps`
- при `refine`
- при отдельных review-сценариях, если статья явно относится к сервису

## Что сюда класть

- бизнес- и функциональные требования
- внутренние и внешние интеграции
- нефункциональные требования
- документацию API, событий и контрактов
- схемы, архитектурные и инфраструктурные материалы

## Рекомендуемая структура

```text
docs/services/<service-name>/
  00_service_overview.md
  20_business_requirements.md
  90_functional_requirements.md
  50_non_functional_requirements.md
  42_internal_integrations.md
  41_external_integrations.md
  63_kafka_topics.md
  80_deployment_scheme.md
```

## Как это участвует в системе

Если в `task.md` указан блок `Сервис:`, система пытается найти подходящие материалы в `docs/services/<service-name>/` и подтягивает их в контекст автоматически.

Это особенно полезно, когда:

- задача короткая и в `task.md` мало деталей
- часть истины уже лежит в сервисной документации
- нужно не только собрать новый draft, но и сверить статью с уже существующим контекстом сервиса
